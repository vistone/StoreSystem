# 编写实现计划

## 概述

编写全面的实现计划，假设工程师对代码库零上下文、品味可疑。记录他们需要知道的一切：每个任务涉及哪些文件、代码、测试、文档、如何验证。将整个计划拆分为小任务。DRY。YAGNI。TDD。频繁提交。

**保存到：** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

## 任务粒度

每个步骤一个动作（2-5 分钟）：
- "编写失败测试" — 一步
- "运行测试确认失败" — 一步
- "实现最小代码" — 一步
- "运行测试确认通过" — 一步
- "提交" — 一步

## 计划文档头部

```markdown
# [功能名] 实现计划

**目标：** [一句话]

**架构：** [2-3 句话]

**技术栈：** Rust + 本项目的依赖

## 全局约束

[规格中的项目级要求]

---
```

## 任务结构

```markdown
### 任务 N: [组件名]

**文件：**
- 创建: `exact/path/to/file.rs`
- 修改: `exact/path/to/existing.rs:123`

- [ ] **步骤 1: 编写失败测试**
  `cargo test test_name`
  预期: FAIL

- [ ] **步骤 2: 实现最小代码**

- [ ] **步骤 3: 验证通过**
  `cargo test`
  预期: PASS

- [ ] **步骤 4: 提交**
  `git commit -m "feat: description"`
```

## 禁止占位符

永远不要写 "TBD"、"TODO"、"实现后续功能"、"适当的错误处理"——这些都是计划失败。
